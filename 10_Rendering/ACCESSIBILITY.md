# Accessibility

Accessibility defines baseline usability requirements for rendered case materials.

## Purpose

A case should be challenging because of reasoning, not because evidence is unreadable, hidden by poor contrast, or dependent on inaccessible presentation choices.

Accessibility supports fairness and playability.

## Scope

Accessibility applies to:

- text documents
- images
- tables
- maps
- diagrams
- package instructions
- hint material
- facilitator material

## Baseline requirements

Rendered material SHOULD support:

- readable text
- clear hierarchy
- sufficient contrast
- stable page layout
- interpretable tables
- inspectable images
- usable print instructions
- non-color-only clue interpretation

## Color use

Critical clues SHOULD NOT depend only on color unless the package clearly provides redundant support.

If color is essential, the print instructions SHOULD state that color printing is required.

## Tiny details

Evidence-bearing details SHOULD be large enough to inspect in the intended medium.

If a detail is intentionally subtle, it SHOULD still be technically legible when players inspect it carefully.

## Realism balance

Documents may look realistic, aged, scanned, copied, or worn.

Those effects SHOULD NOT make required information unusable unless the loss of readability is an intentional and fair part of the puzzle.

## Normative requirements

Critical information SHOULD be readable in the intended release format.

Accessibility issues SHOULD be captured during rendering validation.

Design choices SHOULD NOT create accidental difficulty unrelated to investigation reasoning.

## Validation questions

- Can players read the required text?
- Are evidence-bearing images inspectable?
- Does grayscale printing break interpretation?
- Are tables and timelines understandable?
- Does realism reduce usability too much?
